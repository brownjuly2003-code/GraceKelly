# Batch 102 — extend dry-run profile-gate to /api/v1/orchestrate
Date: 2026-04-25
Tier: S

## Bug

`POST /api/v1/orchestrate` игнорирует `GRACEKELLY_EXECUTION_PROFILE=dry-run`. При profile=dry-run и без `dry_run:true` в body — endpoint всё равно идёт в live `browser.perplexity` adapter и падает на Playwright timeout. Воспроизводится:
```
GRACEKELLY_EXECUTION_PROFILE=dry-run uvicorn ... :8011
curl -X POST :8011/api/v1/orchestrate -d '{"prompt":"2+2","model":"claude-sonnet-4-6","reliability_level":"quick"}'
→ status:"failed", failure_message:"Browser execution failed: Locator.click: Timeout 5000ms exceeded..."
```

В то же время `/api/v1/smart` с тем же payload работает: `[dry-run] Simulated response for: ...`.

## Root cause

CX в batch-101-b Phase A добавил helper `resolve_effective_dry_run(state, payload.dry_run)` в `src/gracekelly/api/routes/_helpers.py` и применил его в **7 из 8** sync routes: `smart`, `smart_v2`, `consensus`, `debate`, `compare`, `batch`, `pipeline`. **Только `orchestrate.py` пропущен.**

Helper:
```python
def resolve_effective_dry_run(state: AppState, dry_run: bool) -> bool:
    profile = getattr(state, "execution_profile", None)
    return dry_run or getattr(profile, "name", None) == "dry-run"
```

Канонический pattern (например в `smart.py`, `smart_v2.py`):
```python
effective_dry_run = resolve_effective_dry_run(state, payload.dry_run)
adapter, backend = resolve_execution_adapter(state, model_spec, effective_dry_run)
# далее везде использовать effective_dry_run вместо payload.dry_run
```

## Scope

**Один файл**: `src/gracekelly/api/routes/orchestrate.py`. Это task-based handler через `OrchestratorService.submit_snapshot`, payload.dry_run пробрасывается в ~10 мест (multipart-upload handler, list-tasks query, retry-task path и т.п.).

Grep `dry_run` по `orchestrate.py` находит ≈30 hits. Нужно правильно различать:
- **gate'ить** на effective_dry_run: места, где `payload.dry_run` / `dry_run` определяет execution path для **текущего** запроса (creating snapshot, computing execution_mode, прокидывая в service.submit, retry-of-task сабмит).
- **не трогать**: места, где `dry_run` — query-filter / read из БД / поле в response (TaskView, list-tasks `?dry_run=true` фильтр, чтение `task.dry_run`/`snapshot.task.dry_run` для отображения).
