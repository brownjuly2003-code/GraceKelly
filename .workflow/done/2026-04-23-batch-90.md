# Batch 90 — Request budget (Phase 4 final deliverable)
Date: 2026-04-23
Tier: S-M (1 new core module + router + config + tests; 0 live quota)

Context (CX read-first):
- Phase 4 Deliverables (`docs/phased-roadmap.md:95-100`) после batch-89 остался один открытый пункт: **`request budget and concurrency limits`**. Concurrency limits закрыты через `ModelConcurrencyGate` (`src/gracekelly/core/concurrency.py`) + `ModelSpec.concurrency_limit`. Этот батч добавляет budget-часть — **rate-limiting на уровне browser submits** для защиты Perplexity Pro квоты.
- Симптом: smart-decomposition может породить 3-5 browser submits на один task, debate — ещё больше. При MAXIMUM-паттерне на 5 моделях с decomposition легко сжечь 15+ submits за один task. Perplexity Pro хардкап ~300/day, ~12-15/hour средний — квота кончится за 1-2 таска.
- Reference pattern: `ModelConcurrencyGate` (`src/gracekelly/core/concurrency.py`) — чистая реализация Lock + state dict + try_acquire/release. Новый `RequestBudgetTracker` делается по образцу.
- Baseline: HEAD `24ca45c`, 2623 passed / 6 skipped / 0 failed / coverage 97%. ruff/mypy clean. 0 live Perplexity submits.
- Interaction с batch-89 fallback policy: budget exhaustion → `FailureCode.RATE_LIMITED` — **НЕ** в fallback-triggering set из batch-89 (`{AUTH_FAILED, PROVIDER_UNAVAILABLE, TIMEOUT}`). Обоснование: budget — осознанный user signal "stop spending browser quota", не infrastructure failure. Fallback на API обошёл бы лимит, что **противоречит** намерению user'а. Design decision: respect budget, не fallback.
- Budget scope: **только browser submits**. API submits (OpenAI/Mistral/Anthropic) — не учитываются (платные, отдельный биллинг, не в этом батче). `step.backend == ExecutionBackend.BROWSER` — единственный triggering path.

Goal батча: новый `RequestBudgetTracker` (по образцу `ModelConcurrencyGate`) считает browser submits per task + rolling hourly window. При превышении лимитов возвращает `FailureCode.RATE_LIMITED` с details `budget_exceeded_kind: "per_task"|"per_hour"`. Два env lever'а — per-task cap + per-hour cap, default None (unlimited, backward-compat).

    - `try_acquire(self, *, task_id: str) -> BudgetAcquireResult` — возвращает dataclass `BudgetAcquireResult(acquired: bool, reason: str | None, usage: dict[str, int])`. Reason ∈ {None, "per_task", "per_hour"}. `usage = {"task_submits": <int>, "hourly_submits": <int>}`;
