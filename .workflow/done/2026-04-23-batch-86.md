When `model_selection_verified=False` (neither `_button_text_matches_requested` nor `selection_evidence["verified"]`), the adapter did click SOMETHING but we cannot trust it was the requested model. Only the unverified-success branch at `:344` leaks.
- Observed side-effect during batch-79 DEBATE live smoke: exec #1 hit `model_mismatch` only because the `_button_text_matches_requested` path inside `select_model` flipped `model_selection_verified=False` early — if the verification heuristic had silently passed but the UI still showed Sonar, nothing would catch it.

Goal батча: when `select_model` returns a verified=False selection, `actual_label` reflects the UI's best-effort evidence (parsed button-text-after, falling back to menu-snapshot first-line), never blindly echoing `provider_model_id`. `_model_matches_expected` gains real comparison material, so a silent Sonar-instead-of-Claude swap becomes a MODEL_MISMATCH failure instead of a successful-but-wrong execution.
