# Patterns: consensus & INTERSECTIONS

Scope note: this comparison is grounded only in the files explicitly listed in batch `101-a-04`. Where GraceKelly delegates behavior into files outside that scope, the document marks the gap instead of guessing.

## 1. INTERSECTIONS (Orchestrator2)

### Entry point and effective inputs

`INTERSECTIONS` is an API-only mode: the route bypasses the unified orchestrator pipeline and calls `run_intersections(...)` directly when `level == INTERSECTIONS` and no model override is supplied. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:235`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:240`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:241`.

The effective inputs of `run_intersections(...)` are `task_id`, `task`, `mirror_telegram`, and `notify`; the route hard-wires the model list to `FIVE_MODELS` and instantiates `ConsensusAnalyzer(MISTRAL_API_KEY)` inside the handler, so callers do not pass a models list, threshold, or embeddings provider at this layer. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:343`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:350`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:365`.

There is a naming mismatch between comments and the actual configured model list. The docstring and progress text say "4 models", but `FIVE_MODELS` currently contains only `CLAUDE`, `GPT`, and `GEMINI`. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:344`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:347`, `Perplexity_Orchestrator2/config/models.py:13`, `Perplexity_Orchestrator2/config/models.py:14`, `Perplexity_Orchestrator2/config/models.py:15`, `Perplexity_Orchestrator2/config/models.py:26`.

### Algorithm

1. Query each configured model sequentially via `_query_models_sequential(...)`; the helper records progress and stores `model`, `answer`, `success`, `error`, `duration_ms`, and `model_actual` for each entry. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:133`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:143`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:148`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:158`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:164`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:350`.
2. Filter down to successful responses with non-empty answers; if fewer than two remain, fail the task immediately. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:356`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:358`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:359`.
3. Build a `ConsensusAnalyzer`, which in turn constructs a `MistralEmbeddings` client. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:365`, `Perplexity_Orchestrator2/orchestrator/consensus.py:33`, `Perplexity_Orchestrator2/orchestrator/consensus.py:34`.
4. Compute pairwise similarities by embedding every answer and applying cosine similarity to each answer pair. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:369`, `Perplexity_Orchestrator2/orchestrator/consensus.py:114`, `Perplexity_Orchestrator2/orchestrator/consensus.py:124`, `Perplexity_Orchestrator2/orchestrator/consensus.py:129`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:27`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:73`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:102`.
5. Cluster the embedded answers with `AgglomerativeClustering` over a precomputed distance matrix (`1 - similarity_matrix`) using average linkage and `distance_threshold = 1 - 0.85`. Source: `Perplexity_Orchestrator2/orchestrator/consensus.py:64`, `Perplexity_Orchestrator2/orchestrator/consensus.py:75`, `Perplexity_Orchestrator2/orchestrator/consensus.py:77`, `Perplexity_Orchestrator2/orchestrator/consensus.py:79`, `Perplexity_Orchestrator2/orchestrator/consensus.py:80`, `Perplexity_Orchestrator2/orchestrator/consensus.py:81`, `Perplexity_Orchestrator2/orchestrator/consensus.py:84`.
6. Derive `consensus_score` as the share of answers in the top cluster; mark `is_unanimous` when all answers fall into one cluster; mark `needs_debate` when consensus falls below `0.70`. Source: `Perplexity_Orchestrator2/orchestrator/consensus.py:91`, `Perplexity_Orchestrator2/orchestrator/consensus.py:97`, `Perplexity_Orchestrator2/orchestrator/consensus.py:101`, `Perplexity_Orchestrator2/orchestrator/consensus.py:102`.
7. Convert pairwise similarity results into `agreements` when similarity is `>= 0.85` and `disagreements` when similarity is `< 0.70`, then package the final `intersections` payload. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:371`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:379`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:381`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:386`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:394`.

### Default models set

The live `INTERSECTIONS` route uses the `FIVE_MODELS` constant, which currently resolves to three display names: `Claude Opus 4.7`, `GPT-5.4`, and `Gemini 3.1 Pro`. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:350`, `Perplexity_Orchestrator2/config/models.py:13`, `Perplexity_Orchestrator2/config/models.py:14`, `Perplexity_Orchestrator2/config/models.py:15`, `Perplexity_Orchestrator2/config/models.py:26`.

In this mode, `answer` and `final_answer` are explicitly `None`. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:403`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:404`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:405`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:406`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:407`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:408`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:409`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:410`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:411`.

### Edge cases

If `MISTRAL_API_KEY` is missing, `run_intersections(...)` does not guard against it before constructing `ConsensusAnalyzer`. The first embeddings call therefore fails only when the HTTP request is attempted, and the route degrades into `intersections = {"error": ...}` instead of failing the task. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:365`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:400`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:401`, `Perplexity_Orchestrator2/orchestrator/consensus.py:33`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:79`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:82`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:98`.

If one provider falls over, the route still proceeds as long as at least two successful model answers remain; provider failure becomes a per-entry `success=False/error=...` record from `_query_models_sequential(...)`. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:152`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:158`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:161`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:356`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:359`.

If embeddings time out, `MistralEmbeddings._fetch_embeddings(...)` retries up to five times with exponential backoff and then raises `"Mistral API timeout after retries"`, which `run_intersections(...)` catches and converts into `intersections.error`. Source: `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:73`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:77`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:104`, `Perplexity_Orchestrator2/orchestrator/mistral_embeddings.py:108`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:400`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:401`.

## 2. consensus V1 (GraceKelly)

### Entry point and inputs

GraceKelly exposes `POST /api/v1/consensus`, and the route summary explicitly labels it "Run iterative consensus V1". Source: `src/gracekelly/api/routes/consensus.py:51`, `src/gracekelly/api/routes/consensus.py:54`.

The public request schema is `prompt`, `model`, `similarity_threshold`, `consensus_target`, `max_rounds`, `variations_per_round`, `use_confidence_weighting`, and `dry_run`. Defaults are `model="claude-sonnet-4-6"`, `similarity_threshold=0.85`, `consensus_target=0.95`, `max_rounds=5`, `variations_per_round=3`, `use_confidence_weighting=True`, and `dry_run=False`. Source: `src/gracekelly/api/routes/consensus.py:27`, `src/gracekelly/api/routes/consensus.py:30`, `src/gracekelly/api/routes/consensus.py:31`, `src/gracekelly/api/routes/consensus.py:32`, `src/gracekelly/api/routes/consensus.py:33`, `src/gracekelly/api/routes/consensus.py:34`, `src/gracekelly/api/routes/consensus.py:35`, `src/gracekelly/api/routes/consensus.py:36`, `src/gracekelly/api/routes/consensus.py:37`.

The core defaults in `ConsensusConfig` match the route defaults: `similarity_threshold=0.85`, `consensus_target=0.95`, `max_rounds=5`, and `variations_per_round=3`. Source: `src/gracekelly/core/consensus.py:7`, `src/gracekelly/core/consensus.py:8`, `src/gracekelly/core/consensus.py:9`, `src/gracekelly/core/consensus.py:10`, `src/gracekelly/core/consensus.py:11`.

### Algorithm

1. Resolve the requested model and adapter, converting invalid model or adapter selection into HTTP 400. Source: `src/gracekelly/api/routes/consensus.py:70`, `src/gracekelly/api/routes/consensus.py:73`, `src/gracekelly/api/routes/consensus.py:75`, `src/gracekelly/api/routes/consensus.py:78`.
2. Build a single-step `ExecutionPlan` with `quorum=1`, `merge_strategy=FIRST_SUCCESS`, and exactly one `ExecutionStep`. Source: `src/gracekelly/api/routes/consensus.py:80`, `src/gracekelly/api/routes/consensus.py:87`, `src/gracekelly/api/routes/consensus.py:88`, `src/gracekelly/api/routes/consensus.py:89`, `src/gracekelly/api/routes/consensus.py:90`.
3. In `dry_run` mode, execute the same plan `variations_per_round` times and return a synthetic perfect-consensus response (`consensus_score=1.0`, `num_clusters=1`, `needs_debate=False`). Source: `src/gracekelly/api/routes/consensus.py:100`, `src/gracekelly/api/routes/consensus.py:102`, `src/gracekelly/api/routes/consensus.py:117`, `src/gracekelly/api/routes/consensus.py:118`, `src/gracekelly/api/routes/consensus.py:119`, `src/gracekelly/api/routes/consensus.py:124`.
4. Outside `dry_run`, require `state.embeddings_client`; if missing, raise HTTP 503. Source: `src/gracekelly/api/routes/consensus.py:128`, `src/gracekelly/api/routes/consensus.py:129`, `src/gracekelly/api/routes/consensus.py:130`.
5. Build `ConsensusConfig` and `ConsensusExecutionConfig`, then construct `ConsensusExecutor(embeddings_client, exec_config)`. Source: `src/gracekelly/api/routes/consensus.py:132`, `src/gracekelly/api/routes/consensus.py:138`, `src/gracekelly/api/routes/consensus.py:143`.
6. Reuse the same resolved adapter/model in `execute_fn(...)`; adapter failures are converted into placeholder strings instead of raising. Source: `src/gracekelly/api/routes/consensus.py:145`, `src/gracekelly/api/routes/consensus.py:153`, `src/gracekelly/api/routes/consensus.py:154`, `src/gracekelly/api/routes/consensus.py:156`.
7. Run the executor and return the aggregated consensus fields. Source: `src/gracekelly/api/routes/consensus.py:159`, `src/gracekelly/api/routes/consensus.py:164`, `src/gracekelly/api/routes/consensus.py:172`.

### Default model set

V1 is not a multi-model route in the scoped implementation. It defaults to one user-selectable model (`claude-sonnet-4-6`) and then reuses that same single `ExecutionStep` across prompt variations and rounds. Source: `src/gracekelly/api/routes/consensus.py:31`, `src/gracekelly/api/routes/consensus.py:80`, `src/gracekelly/api/routes/consensus.py:88`, `src/gracekelly/api/routes/consensus.py:145`, `src/gracekelly/api/routes/consensus.py:150`.

### Output shape

The response model exposes `consensus_score`, `num_clusters`, `best_response`, `weighted_score`, `total_rounds`, `total_llm_calls`, `needs_debate`, and `top_cluster_size`. Source: `src/gracekelly/api/routes/consensus.py:40`, `src/gracekelly/api/routes/consensus.py:48`, `src/gracekelly/api/routes/consensus.py:164`, `src/gracekelly/api/routes/consensus.py:172`.

At the consensus-core level, `build_consensus_result(...)` defines `needs_debate` as `consensus_score < consensus_target`, and `needs_another_round(...)` keeps iterating only while `needs_debate` is true and `round_number < max_rounds`. ### Edge cases

Embeddings are mandatory for non-`dry_run` V1. Missing embeddings return HTTP 503 rather than falling back to a degraded analysis payload. Source: `src/gracekelly/api/routes/consensus.py:128`, `src/gracekelly/api/routes/consensus.py:129`, `src/gracekelly/api/routes/consensus.py:130`.

Provider failures are absorbed as synthetic response strings by `execute_fn(...)`, so failed calls still enter the response pool unless the whole executor later throws. Source: `src/gracekelly/api/routes/consensus.py:153`, `src/gracekelly/api/routes/consensus.py:154`, `src/gracekelly/api/routes/consensus.py:156`.

The route wraps executor exceptions into HTTP 500. Source: `src/gracekelly/api/routes/consensus.py:158`, `src/gracekelly/api/routes/consensus.py:160`, `src/gracekelly/api/routes/consensus.py:162`.

The clustering implementation itself is not visible in the scoped files. V1 delegates into `ConsensusExecutor`, which delegates into `ConsensusAnalyzer` from an unscoped module. Source: `src/gracekelly/core/consensus_execution.py:8`, `src/gracekelly/core/consensus_execution.py:42`, `src/gracekelly/core/consensus_execution.py:70`.

## 3. consensus V2 (GraceKelly)

### Verdict on whether V2 exists separately

The scoped files do not expose a separate public V2 route. The only concrete HTTP surface is still `/api/v1/consensus`, and that route already uses `ConsensusExecutor`. Source: `src/gracekelly/api/routes/consensus.py:10`, `src/gracekelly/api/routes/consensus.py:11`, `src/gracekelly/api/routes/consensus.py:51`, `src/gracekelly/api/routes/consensus.py:143`, `src/gracekelly/api/routes/consensus.py:159`.

Because the batch explicitly called out `consensus_execution.py` / `ConsensusExecutor` as the possible V2 marker, the safest interpretation is: V2 is an internal execution layer, not a separately versioned client API, in the currently scoped code.

### Internal execution behavior

`ConsensusExecutor` accepts an `EmbeddingsClient` plus `ConsensusExecutionConfig`, whose defaults are `variations_per_round=3` and `use_confidence_weighting=True` around a nested `ConsensusConfig`. Source: `src/gracekelly/core/consensus_execution.py:14`, `src/gracekelly/core/consensus_execution.py:16`, `src/gracekelly/core/consensus_execution.py:17`, `src/gracekelly/core/consensus_execution.py:39`, `src/gracekelly/core/consensus_execution.py:40`.

Each round, the executor generates prompt variations from one input prompt, calls the injected `execute_fn` sequentially for each variation, appends the results to a growing `all_responses` list, and re-analyzes the full corpus after every round. Source: `src/gracekelly/core/consensus_execution.py:57`, `src/gracekelly/core/consensus_execution.py:59`, `src/gracekelly/core/consensus_execution.py:63`, `src/gracekelly/core/consensus_execution.py:65`, `src/gracekelly/core/consensus_execution.py:69`, `src/gracekelly/core/consensus_execution.py:70`.

The stop condition is still `needs_another_round(...)`, so the loop exits when the top cluster crosses `consensus_target` or `max_rounds` is exhausted. Source: `src/gracekelly/core/consensus_execution.py:72`, `src/gracekelly/core/consensus.py:36`, `src/gracekelly/core/consensus.py:39`, `src/gracekelly/core/consensus.py:59`.

The executor then chooses `best_response` by `top_cluster.centroid_index`, and optionally computes a `weighted_score` from extracted confidence markers plus `weighted_vote(...)`. Source: `src/gracekelly/core/consensus_execution.py:75`, `src/gracekelly/core/consensus_execution.py:76`, `src/gracekelly/core/consensus_execution.py:78`, `src/gracekelly/core/consensus_execution.py:80`, `src/gracekelly/core/consensus_execution.py:81`, `src/gracekelly/core/consensus_execution.py:87`, `src/gracekelly/core/consensus_execution.py:93`.

### Pattern-layer signals that look more like a future/public V2 family

The pattern registry declares `ExecutionPattern.CONSENSUS` as a 3-model, consensus-enabled, non-iterative pattern and `ExecutionPattern.MAXIMUM` as a 5-model, consensus-enabled, iterative pattern. Source: `src/gracekelly/core/patterns.py:88`, `src/gracekelly/core/patterns.py:90`, `src/gracekelly/core/patterns.py:94`, `src/gracekelly/core/patterns.py:96`, `src/gracekelly/core/patterns.py:97`, `src/gracekelly/core/patterns.py:99`, `src/gracekelly/core/patterns.py:101`, `src/gracekelly/core/patterns.py:105`, `src/gracekelly/core/patterns.py:107`, `src/gracekelly/core/patterns.py:108`.

The resolver maps `ReliabilityLevel.HIGH` to `ExecutionPattern.CONSENSUS` and `ReliabilityLevel.MAXIMUM` to `ExecutionPattern.MAXIMUM`; when resolving from a pattern, `iterative=True` forces MAXIMUM semantics, while `use_consensus=True` without iteration maps to HIGH semantics. Source: `src/gracekelly/core/pattern_resolver.py:29`, `src/gracekelly/core/pattern_resolver.py:32`, `src/gracekelly/core/pattern_resolver.py:33`, `src/gracekelly/core/pattern_resolver.py:56`, `src/gracekelly/core/pattern_resolver.py:63`, `src/gracekelly/core/pattern_resolver.py:76`, `src/gracekelly/core/pattern_resolver.py:83`, `src/gracekelly/core/pattern_resolver.py:85`, `src/gracekelly/core/pattern_resolver.py:86`.

That declarative layer is closer to Orchestrator2's multi-model consensus family than the current V1 route is, but the scoped files do not include the concrete execution path that would prove how those 3-model and 5-model declarations are actually carried out.

## 5. Semantic equivalence?

No: `INTERSECTIONS` and GraceKelly `consensus` are not functional equivalents in the scoped implementation.

`INTERSECTIONS` is a cross-model analysis mode that gathers several model answers, measures pairwise similarity, clusters them, and returns agreement/disagreement metadata without synthesizing a final answer. Source: `Perplexity_Orchestrator2/api/routes/orchestrate.py:350`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:369`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:386`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:409`, `Perplexity_Orchestrator2/api/routes/orchestrate.py:410`.

GraceKelly V1, by contrast, is a single-model self-consistency loop over prompt variations: it reuses one resolved execution step, keeps iterating until `consensus_target` or `max_rounds`, and returns one `best_response`. Source: `src/gracekelly/api/routes/consensus.py:80`, `src/gracekelly/api/routes/consensus.py:88`, `src/gracekelly/api/routes/consensus.py:145`, `src/gracekelly/core/consensus_execution.py:57`, `src/gracekelly/core/consensus_execution.py:72`, `src/gracekelly/core/consensus_execution.py:76`.

The declarative GraceKelly pattern family (`CONSENSUS` = 3 models, `MAXIMUM` = 5 models, iterative) is semantically closer to Orchestrator2's multi-model consensus loop than the current V1 route is, but that execution path is not directly visible in the allowed files. Source: `src/gracekelly/core/patterns.py:90`, `src/gracekelly/core/patterns.py:101`, `src/gracekelly/core/pattern_resolver.py:32`, `src/gracekelly/core/pattern_resolver.py:33`.

The key semantic gap is therefore: `INTERSECTIONS = cross-model agreement/disagreement analysis payload`, while `GraceKelly consensus V1 = single-model iterative self-consistency executor returning one best answer`.
