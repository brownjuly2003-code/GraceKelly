Что такое RAG и зачем он нужен

RAG (Retrieval-Augmented Generation) — подход, при котором LLM (языковая модель) не придумывает ответ из своих весов, а сначала ищет релевантные фрагменты в базе знаний, и только потом генерирует ответ строго на основе найденного.

Зачем это нужно?

Без RAG модель "галлюцинирует" — придумывает правдоподобные, но ложные факты. RAG решает это: модель видит реальные документы и отвечает только по ним. Если нужной информации нет — честно говорит об этом.

Наша система реализует RAG в три уровня сложности, каждый следующий улучшает качество:

Вопрос → Transform → Retrieve → Grade → Generate → Evaluate → Ответ

Level 1: Hybrid Search + Reranking

Базовый уровень решает главную задачу: находить правильные документы и генерировать точный ответ.

BGE-M3 Embeddings

BAAI/bge-m3 Level 1

Мультиязычная модель эмбеддингов (570M параметров, 1024 измерения). Поддерживает 100+ языков, включая русский. Превращает текст в вектор чисел, так что похожие по смыслу тексты имеют близкие векторы.

Время загрузки: ~12 секунд на CPU. Модель кэшируется после первой загрузки.

BM25 Keyword Retrieval

BM25 (Best Matching 25) — классический алгоритм текстового поиска. Он ищет по точному совпадению слов, учитывая частоту слова в документе и редкость слова в коллекции.

Когда BM25 лучше векторного поиска
Запрос: "ошибка E20" — BM25 найдёт документы с точной строкой "E20", тогда как векторный поиск может вернуть документы про "ошибки" в целом.

Reciprocal Rank Fusion (RRF)

RRF объединяет результаты BM25 и векторного поиска в один ранжированный список:

RRF_score(d) = Σ 1 / (k + ranki(d)) где k = 60 (константа сглаживания), ranki(d) — позиция документа d в i-м списке
Пример RRF
Документ "E20" стоит на 1-м месте в BM25 и на 5-м в векторном поиске:
RRF = 1/(60+1) + 1/(60+5) = 0.0164 + 0.0154 = 0.0318
Документ "перегрев" на 3-м месте в BM25 и 1-м в векторном:
RRF = 1/(60+3) + 1/(60+1) = 0.0159 + 0.0164 = 0.0323 ← выше!

Cross-Encoder Reranker

После RRF топ-20 документов перескоряются с помощью cross-encoder/ms-marco-MiniLM-L-6-v2. В отличие от bi-encoder (BGE-M3), cross-encoder видит пару (вопрос, документ) целиком и даёт более точную оценку релевантности. Итоговый топ-5 отдаётся модели.

Level 2: Corrective & Self-RAG

Второй уровень добавляет "самопроверку" на каждом этапе: фильтрация плохих документов, оценка качества ответа, повторная попытка при низком качестве.

Query Transformation

Трансформация запроса Level 2

Прежде чем искать, LLM переформулирует вопрос пользователя в оптимальный поисковый запрос:

  • Убирает разговорный стиль
  • Добавляет синонимы и альтернативные термины
  • Расширяет аббревиатуры
  • Коды ошибок (E20, E21) оставляет как есть

Document Grading (Corrective RAG)

После retrieval каждый документ проверяется LLM на релевантность вопросу. Модель отвечает YES или NO. Нерелевантные документы отбрасываются, чтобы не "отравлять" контекст мусором.

Self-RAG Retry Loop

После генерации ответа LLM сама себя оценивает по шкале 1-100. Если оценка ниже порога (по умолчанию 70), запрос переформулируется и весь цикл повторяется.

Generate → Evaluate (1-100) → score < 70? → Rewrite & Retry

Максимум 2 попытки (настраиваемо). Если после всех попыток оценка низкая — ответ эскалируется к человеку (route: "human").

Semantic Chunking

Вместо разбиения текста по фиксированному количеству символов, semantic chunking анализирует смысловые переходы между предложениями. Граница чанка ставится там, где косинусное сходство между эмбеддингами соседних предложений резко падает.

Как работает

1. Каждое предложение превращается в эмбеддинг (BGE-M3)

2. Считается cosine similarity между соседними предложениями

3. Если similarity < порога — здесь граница чанка

4. Результат: чанки, где каждый охватывает одну логическую тему

Chunk Size Optimizer

Размер чанка — один из самых важных гиперпараметров RAG-системы. Слишком маленький чанк теряет контекст, слишком большой — размывает релевантность. Оптимальный размер подбирается экспериментально по метрикам, а не угадывается.

Почему нельзя просто задать размер "на глаз"?

Оптимальный chunk_size зависит от: языка текста, длины типичных абзацев, размера контекстного окна модели, специфики вопросов. Для русского технического текста он обычно в диапазоне 300-700 символов, но конкретное значение определяется только метриками.

Chunk Size Optimizer перебирает сетку значений и для каждого замеряет три метрики:

chunk_sizes = [300, 400, 500, 600, 700, 900, 1200] overlap_ratio = 0.25 (overlap = chunk_size * 0.25)

Метрики качества: подробный разбор

Каждая метрика измеряет отдельный аспект качества retrieval. Вместе они дают полную картину.

Recall@k (полнота)

Что измеряет

Какую долю релевантных документов мы нашли?

Если по данному вопросу в базе есть 4 подходящих чанка, а retriever вернул 3 из них — Recall@5 = 3/4 = 0.75

Recall@k = |{релевантные документы в топ-k}| / |{все релевантные документы в базе}|
Пример: Recall@5
Вопрос: "Как устранить ошибку E20?"
В базе 3 релевантных документа: [errors_e20.txt, troubleshooting.txt, manual_ch5.txt]
Retriever вернул топ-5: [errors_e20.txt, faq.txt, troubleshooting.txt, pricing.txt, warranty.txt]
Из 3 релевантных найдены 2 (errors_e20, troubleshooting)
Recall@5 = 2/3 = 0.667

Почему Recall@k важнее других метрик (вес 0.6)

Если релевантный документ не найден вообще, модель не сможет ответить — никакие последующие этапы (grading, generation) это не исправят. Пропуск документа — фатальная ошибка. Поэтому Recall имеет наибольший вес.

MRR (Mean Reciprocal Rank)

Что измеряет

Насколько высоко в списке стоит первый правильный документ?

Если первый релевантный документ на 1-й позиции — MRR = 1.0. На 3-й — MRR = 0.33.

MRR = 1 / rank(первого релевантного документа) Для N вопросов: MRR = (1/N) × Σ 1/ranki
Пример: MRR
Вопрос 1: первый релевантный на позиции 1 → RR = 1/1 = 1.0
Вопрос 2: первый релевантный на позиции 3 → RR = 1/3 = 0.333
Вопрос 3: первый релевантный на позиции 2 → RR = 1/2 = 0.5
MRR = (1.0 + 0.333 + 0.5) / 3 = 0.611

Почему MRR на втором месте (вес 0.3)

MRR показывает, насколько быстро пользователь (или модель) найдёт ответ. Если первый правильный документ на 1-м месте, модель сразу получает лучший контекст. Если на 5-м — контекст "разбавлен" нерелевантными фрагментами.

Precision@k (точность)

Что измеряет

Какая доля возвращённых документов действительно релевантна?

Если из 5 возвращённых документов 3 релевантны — Precision@5 = 3/5 = 0.6

Precision@k = |{релевантные документы в топ-k}| / k
Пример: Precision@5
Retriever вернул 5 документов: [errors_e20.txt, faq.txt, troubleshooting.txt, pricing.txt, warranty.txt]
Релевантные: errors_e20.txt, troubleshooting.txt (2 из 5)
Precision@5 = 2/5 = 0.4

Почему Precision@k на третьем месте (вес 0.1)

Низкая Precision означает, что модель получает "мусорные" документы в контексте. Это не так критично, потому что: (а) Document Grading (Level 2) отфильтрует мусор, (б) хорошая модель проигнорирует нерелевантный контекст. Но при очень низкой Precision контекстное окно засоряется, что ухудшает качество ответа.

Composite Score

Три метрики объединяются в одну оценку с настраиваемыми весами:

Composite = 0.6 × Recall@k + 0.3 × MRR + 0.1 × Precision@k

Optimizer перебирает все конфигурации (chunk_size, overlap) из сетки, для каждой вычисляет Composite Score на тестовом наборе, и выбирает конфигурацию с наивысшим баллом.

Победитель: chunk_size=400, overlap=100 (Composite = 0.892)

Миф: "Размер чанка должен быть степенью двойки"

Степени двойки (128, 256, 512, 1024) — это конвенция из мира GPU, где выравнивание памяти имеет значение. Для текстовых чанков нет никакого преимущества у степеней двойки. Текст — не бинарные данные. Chunk_size=400 ничуть не хуже 512, а часто лучше, потому что 400 символов ближе к типичной длине параграфа в русском техническом тексте.

Level 3: Advanced Retrieval

Contextual Retrieval

Метод Anthropic Level 3

Каждый чанк дополняется автоматически сгенерированным заголовком, описывающим его место в документе. Это даёт +49% к точности retrieval (по данным Anthropic).

До и после
Без контекста: "Проверьте уровень охлаждающей жидкости и убедитесь, что вентилятор работает."

С контекстом: "[Документ: Руководство по устранению ошибок. Секция: Ошибка E20 - перегрев двигателя. Этот фрагмент описывает шаги по устранению причин перегрева.]
Проверьте уровень охлаждающей жидкости и убедитесь, что вентилятор работает."

Parent Document Store

Проблема: маленькие чанки точнее находятся, но содержат мало контекста. Большие чанки дают контекст, но хуже ищутся.

Решение: индексируем маленькие чанки (child, ~200 символов), но при нахождении возвращаем большой родительский чанк (parent, ~1000 символов). Точность поиска + широта контекста.

Multi-Query Retrieval

Сложный вопрос разбивается на 2-3 подвопроса. Для каждого выполняется отдельный поиск. Результаты объединяются через RRF.

Пример декомпозиции
Вопрос: "Что делать при ошибке E20 и покрывает ли это гарантия?"
Подвопрос 1: "ошибка E20 устранение перегрев"
Подвопрос 2: "гарантия покрытие ремонт перегрев"

Conversation Memory

Класс ConversationSession хранит историю диалога (до 10 сообщений). Уточняющие вопросы ("А покрывает ли это гарантия?") автоматически раскрываются в полноценный запрос с учётом предыдущего контекста.

RAGAS Evaluation

Система оценки качества RAG-пайплайна по 4 метрикам (реализовано без зависимости от пакета ragas):

МетрикаЧто измеряетМетод
FaithfulnessОтвет использует только факты из контекста?Доля предложений ответа, подтверждённых контекстом
Answer RelevancyОтвет отвечает на вопрос?Доля ключевых слов вопроса, найденных в ответе
Context PrecisionНайденные документы релевантны?Взвешенное (по рангу) перекрытие keywords
Context RecallВсе нужные факты найдены?Доля expected_keywords в контексте

При наличии LLM метрики Faithfulness и Answer Relevancy используют модель для более точной оценки, с автоматическим fallback на keyword-based метод.
